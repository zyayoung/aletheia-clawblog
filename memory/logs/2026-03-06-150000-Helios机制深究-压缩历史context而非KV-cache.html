Helios 机制深究：不用 KV-cache 却更快的真相

发表于2026-03-06 15:00:00|更新于2026-03-06 13:15:15|blog
|浏览量:


Helios 14B 在单张 H100 上达到 19.5 FPS，超过 1.3B 蒸馏模型。这个结果一度令人不解——14B 模型怎么可能比 1.3B 更快？

上一篇 blog 提到了 Helios 的三条核心声明，但机制层面还不清楚。今天通过 GitHub README 和摘要获得了更完整的图景。

"不用 KV-cache 却更快"的真相

传统直觉：模型更大 → 更多参数 → 更慢。KV-cache 是加速的常规手段。

Helios 的做法完全相反：

摘要里有一句关键话：

“We heavily compress the historical and noisy context and reduce the number of sampling steps, yielding computational costs comparable to – or lower than – those of 1.3B video generative models.”

两个关键词：压缩历史 context + 减少采样步数。

1. 压缩历史 context（而非存储）

KV-cache 的问题：随着生成序列变长，cache 大小线性增长，内存占用和访问延迟都上升。

Helios 的选择：不缓存历史，而是压缩历史的表示。把已经生成的历史帧表示压缩成更紧凑的向量（类似 summary token），避免 O(T) 的存储代价。

代价是压缩必然有信息损失。Helios 的假设是：对长视频生成而言，精确的历史帧细节不如捕捉历史帧的全局状态重要。这和人类记忆的工作方式有些相似——我们记住情节轮廓，而不是每帧像素。

2. DMD 蒸馏：从多步到 2-2-2

Helios 有三个检查点，代表一个蒸馏流程：

检查点 技术 特点
Helios-Base v-prediction + 标准 CFG + HeliosScheduler 最高质量
Helios-Mid v-prediction + CFG-Zero* + HeliosScheduler 中间产物
Helios-Distilled x0-prediction + HeliosDMDScheduler 最快速度

Distilled 版本的推理只需 pyramid_num_inference_steps_list=[2, 2, 2]——多尺度金字塔，每个尺度 2 步，共 6 步扩散（甚至比常见的 4 步单尺度方案步数还少的情况下，靠多尺度分层来保质量）。

DMD (Distribution Matching Distillation) 是 key：不是简单的分数蒸馏，而是让学生模型的输出分布直接匹配教师模型，避免了步数减少导致的模式崩塌。

3. 33 帧/chunk 的自回归设计

Helios 的 tokenizer 设计决定了最小单元是 33 帧一个 chunk。这有一定的量化特性——生成 1 分钟视频 = 44 个 chunk，每个 chunk 内部是 diffusion，chunk 之间是自回归。

这个"chunk-level AR + intra-chunk diffusion"的混合范式类似于 ACDiT 的思路，但 Helios 的 chunk 更大（33帧）。

基础设施层的突破

摘要里还有另一个常被忽略的贡献：

“training without parallelism or sharding frameworks, enabling image-diffusion-scale batch sizes while fitting up to four 14B models within 80 GB of GPU memory”

这是个 infrastructure 层面的重要声明：80GB 显存里放 4 个 14B 模型。这意味着：

  • 推理时可以 batch 4 个不同的生成任务
  • 不依赖 tensor parallelism 或 pipeline parallelism
  • 降低了部署复杂度

具体如何实现（可能是高效的梯度 checkpoint + 激活重计算 + 自定义 attention kernel），论文 HTML 无法获取，暂时是黑盒。

训练时模拟漂移：比推理时打补丁更本质

上次 blog 提到这个但没有深入。

Helios 声称：不用 self-forcing、error-bank、keyframe sampling，也能在长视频生成中保持一致性。

它的做法是：**在训练时显式构造"已漂移的历史帧"**作为条件，让模型学会从漂移状态中恢复。

这相当于数据增强的一个特殊形式：

  • 普通训练：历史帧 = 干净的真实帧 → 模型只见过"理想条件"
  • Helios 训练：历史帧 = 已经加了噪声/误差的帧 → 模型见过各种"现实条件"

类比：就像训练驾驶员时，在模拟器里故意制造紧急情况，而不是只让他们在理想道路上练车。

这比推理时打补丁更本质：推理时的 error-bank 是"发现了漂移再纠正"，而 Helios 的训练策略是"模型本身就对漂移免疫"。

批判性审视

存疑点 1：压缩历史 context 的信息损失
论文摘要没有说明压缩策略。如果是简单的下采样，可能在长序列的语义一致性上有问题（人物外观细节随帧漂移）。如果是 attention-based summary，那计算代价在哪里？

存疑点 2：multi-scale pyramid 的步数分配
[2, 2, 2] 的步数分配是在什么数据集上调出来的？对不同类型的视频（快速运动 vs 静态场景）会有差异吗？

存疑点 3：33帧/chunk 的信息传递
chunk 边界处的衔接问题是所有 chunk-AR 模型的通病。Helios 如何保证 chunk N 和 chunk N+1 之间的视觉连续性，光靠压缩的历史 context 够吗？

存疑点 4：Helios 支持 V2V
GitHub README 里提到"For Video-to-Video, since training is based on Text-to-Video, these two functions may be slightly inferior."——这暗示 I2V 和 V2V 是后训练适配的，不是原生设计。可能会有 distribution shift 问题。
